# Receipt Inbox SaaS 프로젝트 핵심 정리

## 문서 목적
이 문서는 새로운 대화창에서 동일한 프로젝트를 정확히 이어가기 위해, 지금까지 합의된 제품 컨셉, 비즈니스 방향, UX 원칙, 시스템 구조, 확장 전략을 최대한 빠짐없이 정리한 프로젝트 기준 문서이다.

이 문서에서 가장 중요한 것은 다음 두 가지다.

1. 이 제품은 더 이상 단순한 출장경비 리포트 도구가 아니다.
2. 이 제품은 영수증 업로드부터 리포트 생성까지 이어지는 Receipt Inbox SaaS로 재정의되었다.

따라서 이후 어떤 대화나 설계, 코드 작업도 이 문서 기준을 우선 참조해야 한다.

---

# 1. 최종 제품 컨셉

## 1-1. 제품의 본질
최종적으로 합의한 제품의 본질은 다음과 같다.

Receipt → Structured Data → Report

즉, 사용자가 영수증 사진을 업로드하면 AI/OCR이 영수증 데이터를 추출하고, 이를 구조화된 데이터로 저장한 뒤, 사용자가 원하는 기간과 양식에 맞는 리포트를 생성하는 구조이다.

이 제품은 단순 OCR 도구가 아니며, 단순 출장비 정산 앱도 아니다.

최종 포지셔닝은 다음과 같다.

Receipt Inbox SaaS

또는 보다 기능적으로 설명하면,

Receipt Processing + Report Generation Platform

## 1-2. 기존 컨셉에서의 전환
처음에는 출장경비 리포트 자동 생성 도구 성격이 강했으나, 현재는 그보다 더 넓은 개념으로 변경되었다.

즉,

- 출장경비 리포트는 여러 리포트 유형 중 하나의 모드일 뿐이다.
- 핵심은 영수증을 먼저 쌓아두고, 나중에 주간/월간/출장/사용자 정의 기간 기준으로 리포트를 만드는 구조이다.
- 따라서 제품의 중심은 Trip Expense가 아니라 Receipt Inbox이다.

## 1-3. 핵심 사용자 가치
이 제품의 핵심 가치는 다음과 같다.

1. 사진 찍어서 업로드하면 된다.
2. 영수증은 쌓아둘 수 있다.
3. 필요할 때 검토 후 리포트를 생성한다.
4. 리포트는 회사/회계용 양식으로 출력된다.
5. AI가 최대한 도와주되, 최종 검증은 사용자가 한다.

중요한 UX 문장으로 정리하면 다음과 같다.

사진 찍기 → 자동 정리 → 검토 → 리포트 생성

# 2. 비즈니스 방향

## 2-1. B2B 우선 전략
현재 우선순위는 B2B이다.

즉,

- 회계펌 채널
- 회계펌이 관리하는 고객사
- 회사 직원/관리자

를 중심으로 먼저 시작한다.

직접 일반 소비자 대상으로 크게 공개하는 방향은 초기 우선순위가 아니다.

## 2-2. 회계펌과의 관계
회계펌은 이 제품의 직접 사용자라기보다, 고객사에게 생산성 향상 서비스를 제공하는 파트너 채널로 본다.

핵심 방향은 다음과 같다.

- 회계펌이 고객사에게 이 도구를 부가 서비스처럼 제공한다.
- 회계펌은 "우리는 단순히 숫자만 정리하는 것이 아니라, 고객사의 실제 업무 생산성을 높여준다"는 메시지로 활용한다.
- 따라서 이 제품은 회계펌이 고객사에게 판매하는 독립 제품이라기보다는, 서비스 경쟁력을 높이는 도구로 활용된다.

## 2-3. 장기 확장
향후에는 개인사업자 및 일반 사용자에게도 확장 가능성이 있다.

예시 방향:

- 앱스토어 등록
- 개인용 구독 모델
- 월간 소비 리포트 생성
- 개인사업자용 비용 정리

다만 이 영역은 현재 즉시 추진 대상이 아니라 Phase 2 이후의 확장 영역으로 본다.

# 4. 제품 핵심 구조

## 4-1. 시스템 개념 구조
권장 구조는 다음과 같다.

Client (Web / Mobile)
↓
API Server
├ OCR Processing
├ Receipt Database
├ Report Engine
└ Object Storage

즉,

1. 사용자가 Web 또는 Mobile에서 업로드한다.
2. API 서버가 파일을 받는다.
3. OCR Processing이 영수증 데이터를 추출한다.
4. Receipt Database에 구조화된 데이터가 저장된다.
5. 이미지 자체는 Object Storage에 저장한다.
6. Report Engine이 리포트 생성 시 이 데이터를 가져다 쓴다.

## 4-2. 이미지 저장 전략
이미지는 서버 로컬 저장이 아니라 Object Storage를 기본으로 한다.

추천 예시:

- AWS S3
- Cloudflare R2
- Backblaze B2

DB에는 이미지 자체를 저장하지 않고, 다음 같은 참조 정보만 저장한다.

- image_url
- thumbnail_url

이 방향은 향후 저장 비용과 서버 운영 측면에서 유리하다.

# 6. Receipt Upload UX

## 6-1. PC
PC에서는 파일 업로드 기반이다.

추천 방식:

- 파일 선택
- Drag & Drop
- 다중 파일 업로드

## 6-2. Mobile
모바일에서는 다음 두 가지가 필요하다.

- 카메라 촬영
- 갤러리 선택

즉, 모바일은 나중에 확장하되, 처음부터 Mobile Upload API 관점은 열어두는 것이 좋다.

## 6-3. Bulk Upload
여러 장을 한 번에 올릴 수 있는 Bulk Upload는 반드시 필요하다.

왜냐하면 사용자는 보통 영수증을 한 장씩만 올리지 않고, 한 번에 여러 장 정리하는 경우가 많기 때문이다.

# 7. Receipt Inbox 구조

## 7-1. Inbox 개념
제품의 중심은 Receipt Inbox이다.

즉, 영수증은 바로 리포트로 가는 것이 아니라 먼저 Inbox에 쌓인다.

예시 흐름:

영수증 업로드
→ OCR 처리
→ Receipt DB 저장
→ Inbox에서 Review
→ 이후 Report 생성

## 7-2. Receipt Status
각 영수증에는 상태값이 있어야 한다.

예:

- processed
- needs_review
- duplicate

이 상태는 검색과 리포트 생성 전 검토 UX에서 중요하다.

## 7-3. Report Status
리포트와의 연결 여부도 구분되어야 한다.

예:

- assigned
- unassigned

즉, 아직 리포트에 포함되지 않은 영수증을 따로 볼 수 있어야 한다.

# 9. 리포트 생성 방식

## 9-1. 기본 원칙
자동 리포트 생성은 하지 않는다.

그 이유는 다음과 같다.

- AI가 100% 정확하지 않다.
- 영수증 누락 가능성이 있다.
- 잘못된 OCR/카테고리/중복이 있을 수 있다.
- 사용자의 검증 단계가 반드시 필요하다.

## 9-2. 권장 방식
자동 생성 대신, 일정 주기에 이메일 알림을 보내고 사용자가 최종 검토 후 리포트를 생성하는 구조로 간다.

권장 흐름:

기간 종료
→ 이메일 알림
→ 사용자 로그인
→ Receipt Review
→ Final Report Generate
→ Submit

## 9-3. 이메일 알림 예시
이메일은 단순 알림이 아니라 검토 포인트를 같이 보여주는 것이 좋다.

예:

- Receipts found: 32
- Uncategorized: 3
- Possible duplicates: 1

그리고 버튼:

- Review Receipts
- Generate Report

이 구조는 사용자가 바로 무엇을 해야 하는지 이해하게 해 준다.

## 9-4. 향후 Mobile
향후 모바일 앱이 생기면 이메일 외에 앱 푸시 알림도 가능하다.

예:

- Weekly report ready for review
- Monthly report ready for review

# 10. Report Templates

## 10-1. 기본 방향
관리자 페이지에서 리포트 양식을 직접 자유롭게 만드는 구조로 가지 않는다.

대신 시스템이 제공하는 여러 템플릿 중에서 활성화하는 방식으로 간다.

즉,

System Templates
- Weekly Expense
- Monthly Expense
- Trip Expense
- Vendor Summary
- Category Summary

관리자는 고객사에 맞게 이 템플릿을 활성화 또는 비활성화한다.

## 10-2. 고객 맞춤형 양식
고객별 완전 커스터마이징은 가능하되, 기본 제품 기능이 아니라 별도 비용을 받고 진행하는 방향으로 둔다.

즉,

- 기본 제품: 시스템 템플릿 활성화 방식
- 추가 유료 서비스: 고객 맞춤형 양식/매핑 개발

이렇게 나누는 것이 맞다.

# 11. Branding Settings

## 11-1. 목적
템플릿 구조는 고정하되, 기본 브랜딩 수준의 커스터마이징은 허용한다.

추천 Admin 설정:

- company_name
- company_logo
- report_footer
- default_currency

## 11-2. 적용 방식
리포트 생성 시 다음 요소를 자동 반영한다.

- 리포트 헤더
- 회사명
- 회사 로고
- 푸터 문구

이 구조는 고급스러워 보이면서도 구현 난이도가 과하지 않다.

# 12. Category / Account Mapping

## 12-1. 카테고리의 역할
카테고리는 단순 분류를 넘어 회계 계정 연결의 출발점이다.

예시 카테고리:

- Food
- Travel
- Fuel
- Supplies
- Office

## 12-2. Category → Account Code 매핑
회계 관점에서는 카테고리별로 계정코드를 연결할 수 있어야 한다.

예:

- Food → 6100
- Travel → 6200
- Fuel → 6300

즉, Category Mapping Table이 필요하다.

## 12-3. 리포트 유형별 차이
특히 Trip Expense 같은 리포트에서는 회사가 정한 계정코드나 비용 항목 체계에 맞춰 선택할 수 있어야 한다.

따라서 카테고리 자체와 계정 연결은 분리된 구조가 좋다.

---

# 13. Vendor Auto Classification

Vendor가 쌓일수록 자동 분류가 되게 하는 방향에 합의했다.

예:

- Starbucks → Food
- Shell → Fuel
- Uber → Travel

이 기능은 UX 품질을 크게 끌어올린다.

즉,

Vendor Mapping
- vendor_name
- suggested_category
- account_code(optional)

구조를 두는 것이 좋다.

---

# 14. Duplicate Detection

중복 업로드 감지는 반드시 필요하다.

기본 로직 예시:

- 동일 vendor
- 동일 amount
- 동일 receipt_date

이 경우 possible duplicate로 표시한다.

이 기능은 실제 운영에서 매우 중요하다. 사용자는 같은 영수증을 두 번 올릴 가능성이 높기 때문이다.

---

# 15. 기준 날짜 원칙

리포트와 기간 분류의 기본 기준 날짜는 Receipt Date로 한다.

즉,

- Receipt Date가 주간/월간 리포트 포함 기준이다.
- Upload Date는 참고 정보일 뿐 기준이 아니다.
- Posting Date는 회계 입력 시점에서 별도 개념이다.

따라서 데이터 모델에서는 최소 다음 필드가 필요하다.

- receipt_date
- upload_date
- report_period
- posting_date

예시:

3/31 영수증을 4/2에 업로드했더라도,
기본적으로 3월 리포트에 포함되어야 한다.

# 16. UX 원칙

## 16-1. 핵심 UX 원칙
이 제품은 기능이 많아져도 사용자가 느끼기에는 매우 간단해야 한다.

핵심 경험은 다음과 같아야 한다.

사진 찍기 → 끝에 가까운 경험

즉, 내부적으로는 OCR, 분류, 상태관리, 리포트 구조가 있어도, 사용자가 느끼는 진입 장벽은 낮아야 한다.

## 16-2. Step UI
현재 한 페이지에서 모든 걸 처리하는 구조보다는 단계형 UX가 더 고급스럽고 편리하다.

다만 단계가 많아져서 사용자가 피곤해지면 안 된다.

권장 구조:

- Step 1 Upload
- Step 2 Review
- Step 3 Report

그리고 자동으로 넘어갈 수 있는 곳은 Auto Skip한다.

즉,

- OCR이 문제없이 끝나면 바로 Review 최소화
- 문제가 있거나 리포트 생성 직전일 때만 검토 유도

## 16-3. 로그인 후 모드 분기
로그인 후 사용자가 무엇을 하려는지에 따라 다음 UX가 달라져야 한다.

예:

1. Upload Mode
- 영수증 업로드 중심

2. Report Mode
- 리포트 검토 및 생성 중심

즉, 로그인 후 진입 목적에 따라 화면 흐름이 달라져야 한다.

# 17. 저장소 및 보관 정책

## 17-1. 저장 비용 문제
영수증 이미지를 오래 저장하면 저장소 비용이 늘 수 있다.

하지만 서버 로컬 저장 대신 Object Storage를 사용하면 비용은 상대적으로 낮다.

그래도 제품 차원에서 저장 정책을 두는 것이 좋다.

## 17-2. 권장 정책
예시:

Basic
- 1GB storage
- 12 months retention

추가 옵션
- extra storage
- archive option

즉,

- 기본 저장 공간 제공
- 일정 기간 보관
- 장기 보관이나 추가 공간은 옵션화

## 17-3. 구현 복잡도 고려
초기에는 너무 복잡하게 플랜을 나누기보다,

- Object Storage 구조
- retention_date 개념
- storage_usage 계산 기반

정도만 열어두는 것이 바람직하다.

# 20. 현재 합의된 핵심 원칙 요약

이 프로젝트의 최종 컨셉에서 절대 누락되면 안 되는 핵심은 다음과 같다.

1. 제품은 더 이상 출장경비 리포트 전용이 아니다.
2. 제품의 최종 컨셉은 Receipt Inbox SaaS이다.
3. 영수증은 먼저 쌓아두고, 나중에 기간과 템플릿을 선택하여 리포트를 생성한다.
4. 리포트 자동 생성보다는 이메일 알림 후 사용자 검토/최종 생성 방식이 더 적합하다.
5. AI는 보조이며, 사용자 검증 단계를 반드시 둔다.
6. 관리자 페이지에서는 시스템 템플릿을 활성화하고, 기본 브랜딩(회사명/로고 등)은 지원한다.
7. Category → Account Code 매핑 구조가 필요하다.
8. Vendor 기록이 쌓일수록 자동 분류 정확도를 높이는 방향으로 간다.
9. 이미지 저장은 Object Storage 구조를 기본으로 하며, 보관 정책/용량 정책은 차후 플랜화 가능하다.
10. PC Web을 먼저 완성하고, Mobile Upload는 이후 확장한다.
11. B2B 중심으로 시작하되, 향후 개인사업자/일반 사용자로 확장 가능성을 열어둔다.

# 22. 새 대화창 시작용 전달문

아래 문장을 새 대화창 첫 메시지로 사용하면 된다.

우리는 Receipt Inbox SaaS를 설계 중이다.

최종 컨셉은 더 이상 출장경비 리포트 도구가 아니라,
Receipt → OCR → Receipt Database → Report Generator 구조의 Receipt Inbox SaaS이다.

핵심 원칙은 다음과 같다.
- 영수증은 먼저 쌓아둔다.
- 사용자는 나중에 기간과 템플릿을 선택해 리포트를 생성한다.
- 리포트 자동 생성보다는 이메일 알림 후 사용자 검토/최종 생성 구조가 맞다.
- AI는 보조이고 사용자 검증 단계가 반드시 있어야 한다.
- 관리자 페이지에서는 시스템 템플릿 활성화 + 기본 브랜딩(회사명/로고)을 지원한다.
- 카테고리와 계정코드 매핑이 필요하다.
- Vendor 기록이 쌓일수록 자동 분류를 한다.
- 이미지 저장은 Object Storage 기반이다.
- PC Web 우선, Mobile Upload는 이후 확장한다.
- B2B 우선, 향후 개인사업자/일반 사용자 앱 확장 가능성을 열어둔다.

  - `report.view_all` 또는 운영자는 테넌트 범위에서 생성 가능
